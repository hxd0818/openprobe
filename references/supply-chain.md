# 产业链上下游调研 ⭐ v9.2

> **v9.2 核心变革**：从"固化4个方法（P1-P4）"升级为**"Agent自主推理 + 工具箱辅助"**。
> 不再预设"零售业用什么方法、B2G用什么方法"，
> 而是教会Agent一套**供应链分析思维**，由Agent根据目标企业的实际情况自主规划发现策略。
>
> 适用场景：所有行业、所有企业类型。Agent自主适配。

## 1. 核心思维

### 旧版问题诊断

| 问题 | 表现 | 根因 |
|------|------|------|
| 方法与行业错配 | P1招投标⭐⭐⭐⭐⭐排第一，但对纯消费零售企业效用为0 | 方法优先级写死，不考虑目标企业特征 |
| P4技术足迹浪费篇幅 | DNS/子域名分析对连锁零售无意义 | 照搬IT/SaaS行业的方法论 |
| Agent不做判断 | 机械地按P1→P2→P3→P4顺序执行 | 缺乏"先想再做"的环节 |
| 行业盲区 | 只有B2G/B2B/B2C三级粗分 | 无法应对同一行业内不同商业模式 |

### 新版框架总览

```
Agent 自主推理链:
  
  Step 1: 绘制供应链图谱（"画出来"）
    → 这家公司的上游是谁？下游是谁？中间经过什么？
    → 用文字或图形描述完整的链路结构
    
  Step 2: 识别关键节点（"标重点"）
    → 哪些节点掌握了公司的命脉？（单一依赖/高价值/不可替代）
    → 哪些节点的信息是公开可获取的？哪些是黑箱？
    
  Step 3: 规划发现策略（"想办法"）
    → 针对每个关键节点，Agent自己决定用什么手段获取信息
    → 这是Agent自主决策的核心步骤
    
  Step 4: 执行与调整（"动手+应变"）
    → 执行搜索，根据结果动态调整方向
    → 发现新线索时回溯修正图谱
    
  Step 5: 风险评估（"下结论"）
    → 基于收集到的数据做集中度分析和风险评估
```

**核心区别**：

| | 旧版 (v9.0) | 新版 (v9.2) |
|---|---|---|
| 谁决定方法 | Skill 文件写死 P1-P4 | **Agent 根据 Step 1-2 自主选择** |
| 行业适配 | 无（或预想的有限分类） | **Agent 读数据后自动适配** |
| 工具定位 | "必须执行的步骤" | **"可选用的参考建议"** |
| Agent角色 | 脚本执行者 | **供应链分析师** |

## 2. Step 1：绘制供应链图谱

> 在搜索任何上下游信息之前，Agent 必须先基于 Phase 0 的已有数据（百科+泛化搜索），
> 用自己的推理**画出**目标企业的供应链结构。

### 2.1 供应链五问

| # | 问题 | 回答后得到的信息 |
|---|------|-----------------|
| Q1 | **上游给这家公司提供了什么？** | 原材料 / 产品 / 技术 / 服务 / 资金 / 场地 / 品牌... |
| Q2 | **这些"投入"来自谁？** | 具体的供应商类型（不是具体公司名，而是类别） |
| Q3 | **这家公司把什么给了下游？** | 产品 / 服务 / 体验 / 平台 / 解决方案... |
| Q4 | **下游是谁？他们怎么获得？** | 客户类型 + 渠道类型 + 决策路径 |
| Q5 | **整条链路上谁掌握最大话语权？** | 上游强势? 下游强势? 公司自身有议价权? |

### 2.2 图谱输出格式

Agent 在 Step 1 完成后，**必须输出**以下格式的供应链图谱（自然语言即可，不需要画图）：

```markdown
## [目标企业名] 供应链结构图谱

### 上游（输入端）
┌─────────────────────────────────────────────┐
│ 层级1（直接输入）：                          │
│   • [输入品类A] → 来自 [供应方类型]          │
│   • [输入品类B] → 来自 [供应方类型]          │
│   • [输入品类C] → 来自 [供应方类型]          │
│                                             │
│ 层级2（间接输入）：                          │
│   • [层级1的供应方] 自身需要 [更深层供应]     │
│                                             │
│ 关键特征：                                  │
│   • 上游集中度判断：高/中/低（基于商业模式推断）│
│   • 可替代性：容易/困难（切换成本高低）       │
└─────────────────────────────────────────────┘

                    ▼ 目标企业 ▼

### 下游（输出端）
┌─────────────────────────────────────────────┐
│ 直接客户/渠道：                              │
│   • [客户类型A] 通过 [渠道A] 获得            │
│   • [客户类型B] 通过 [渠道B] 获得            │
│                                             │
│ 最终消费者/用户：                            │
│   • [用户画像]                               │
│                                             │
│ 关键特征：                                  │
│   • 下游集中度判断：高/中/低                  │
│   • 议价权归属：上游主导/下游主导/均衡        │
└─────────────────────────────────────────────┘

### 话语权分布
[描述整条链路的力量分布：谁最强势？谁最弱势？为什么？]
```

### 2.3 示例：KK集团（正确做法）

```markdown
## KK集团 供应链结构图谱

### 上游（输入端）
层级1（直接输入）：
  • 第三方品牌商品 → 来自 各品牌方/代理商（美妆/食品/家居等）
  • 自有品牌商品(OEM/ODM) → 来自 合同制造工厂
  • 门店场地 → 来自 购物中心/商业地产商（租金+业绩扣点）
  • 物流配送 → 可能自建或外包

层级2（间接输入）：
  • OEM工厂需要原材料/设计方案
  • 品牌方自身也有其供应链

关键特征：
  • 上游集中度：低（前五大供应商占比仅7.8%，招股书披露）
  • 可替代性：第三方品牌容易切换；OEM有一定磨合成本；
    商场渠道最难替换（装修投入+租约锁定）

### 下游（输出端）
直接渠道：
  • 自营门店消费者（占比~85%）
  • 加盟门店（占比~15%，历史高峰76%）
最终消费者：
  • Z世代（18-28岁），一二线城市，追求颜值+新鲜感

关键特征：
  • 下游集中度：中低（分散的C端消费者）
  • 但渠道集中度高！依赖购物中心主力店位置
  • 议价权：商场方在强势位置（扣点模式下业绩不达标即清退）

### 话语权分布
商场/地产商 > 品牌方 > KK集团 > 加盟商 > C端消费者
原因：KK的大店模式使其对优质商圈位置高度依赖，
      而2025-2026年的多起闭店事件已证明这种依赖的风险。
```

**注意**：这个图谱是在**任何针对性搜索之前**就画出来的。它来自 Agent 对 KK 商业模式的初步理解。后续搜索的目标是**验证和填充细节**，而非从头发现。

## 3. Step 2：识别关键节点与信息可及性

### 3.1 关键节点标注

在 Step 1 的图谱基础上，标注三类节点：

| 节点类型 | 定义 | 应对方式 |
|---------|------|---------|
| 🔴 **命脉节点** | 出问题会直接重创业务的节点 | **最高优先调查**，必须找到具名实体和量化数据 |
| 🟡 **重要节点** | 影响盈利能力或增长潜力的节点 | 中等优先，尽量找到具名实体 |
| 🟢 **常规节点** | 有用但非关键的节点 | 低优先级，有资源时补充 |

**标注原则**：
- 单一依赖 = 🔴（如只有一个OEM供应商、只依赖一类渠道）
- 高价值投入 = 🔴（如租金占成本比例极高）
- 可替代性强 = 🟢（如普通消费品采购）

### 3.2 信息可及性预判

对每个🔴/🟡节点，Agent 必须**预先判断**信息获取难度：

| 可及性 | 特征 | 策略 |
|--------|------|------|
| **高** | 上市公司年报/招股书/公告/新闻常披露 | 直接搜索即可，预期效果好 |
| **中** | 行业报告/展会/协会/间接提及可能覆盖 | 需要多角度搜索，可能只能获部分信息 |
| **低** | 属于商业机密/非公开信息 | 设法通过间接线索推断（如从诉讼文书/招聘信息/工商变更反推） |
| **极低** | 几乎不可能通过OSINT获取 | **诚实标记为"信息缺口"**，不要编造或猜测 |

### 3.3 输出：优先级矩阵

```markdown
## 上下游调查优先级矩阵

| # | 节点 | 类型(上/下) | 风险等级 | 信息可及性 | 调查优先级 | 计划使用的方法（Agent自填）|
|---|------|------------|---------|-----------|-----------|---------------------|
| 1 | [例: OEM供应商] | 上游 | 🔴 | 中 | P1 | [Agent自填] |
| 2 | [例: 购物中心渠道] | 下游 | 🔴 | 高 | P1 | [Agent自填] |
| 3 | [例: 第三方品牌方] | 上游 | 🟡 | 高 | P2 | [Agent自填] |
```

## 4. Step 3：自主规划发现策略

> **这是整个流程的核心创新点。**
> 
> **Agent 根据Step 1-2的分析结果，自行决定针对每个关键节点用什么方法去发现信息。**
> 
> 第7节的工具箱仅作**参考建议**——Agent 可以使用工具箱中的工具，也可以完全不用而采用自己认为更好的方法。
> 唯一约束：**必须在每轮搜索前声明本轮目标和理由。**

### 3.1 规划要求

Agent 在开始上下游专项搜索前，**必须输出**一段简短的调查计划：

```markdown
## 上下游调查计划

### 基于供应链分析的判断
[用2-3句话说明你对这家公司供应链的核心判断]

### 本轮次规划
**Round X（上下游专项）：**
- 目标节点：[具体要查哪个上下游环节]
- 选择的方法：[具体说明用什么手段]
- 选择理由：[为什么认为这个方法对这个节点有效]
- 预期产出：[希望获得什么信息]
```

### 3.2 自主规划的指导原则（非强制规则）

以下原则帮助 Agent 做出好的规划决策，但**不限制** Agent 的发挥：

**原则1：从信息最可能泄露的地方入手**
- 上市公司/拟上市公司 → 招股书、年报、公告（法定披露要求）
- 有融资历史 → 融资报道、BP（商业计划书）泄露
- 有诉讼/纠纷 → 判决书、仲裁文书（事实披露详尽）
- 有大量门店/加盟 → 加盟招商信息、闭店新闻、商场公示
- 有招投标 → 中标公告（最准确）

**原则2：从已知实体向外扩散**
- 已知上游某类供应商 → 搜该类供应商的客户名单 → 反向确认
- 已知下游某类客户 → 搜该客户的供应商名单 → 发现同行
- 已知竞争对手 → 参考竞对的公开供应链信息（年报/招股书/采访）

**原则3：不同类型的"证据源"互补使用**

| 证据源 | 擅长提供 | 局限性 |
|--------|---------|--------|
| 招股书/年报 | 最准确的财务数据和合作关系 | 仅适用于上市/拟上市公司 |
| 新闻报道 | 最新动态、事件细节 | 可能夸大或不准确 |
| 招投标公告 | 确定的交易关系和金额 | 仅限G端/B2B |
| 诉讼文书 | 隐藏的商业关系和细节 | 需要知道去搜什么 |
| 招商/加盟信息 | 渠道模式和合作伙伴 | 可能过时或宣传性质 |
| 社交媒体/评论 | 一线运营真实反馈 | 噪音大，需甄别 |
| 工商变更记录 | 股权/高管变动痕迹 | 信息片段化 |
| 行业展会/协会 | 行业玩家名录 | 可能不全 |

**原则4：动态调整优于死守计划**

```
执行过程中:
  ├─ 发现了预料之外的重要线索 → 追踪它，哪怕偏离原计划
  ├─ 某个节点确实搜不到任何信息 → 标记为"信息缺口"，转向其他节点
  ├─ 某个方法连续2轮无效 → 换方法，不要死磕
  └─ 发现新的供应链关系（如原本不知道的关联方）→ 回溯更新 Step 1 的图谱
```

## 5. Step 4：执行与动态调整

### 5.1 每轮执行的记录格式

每轮上下游搜索结束时，Agent 更新以下追踪表：

```markdown
### Round X 执行记录
- **目标节点**: [本轮查的是什么]
- **使用方法**: [实际用了什么]
- **结果摘要**: [发现了什么，关键数据点]
- **置信度提升**: [哪些节点的认知有提升]
- **下一步**: [还缺什么，下一轮怎么补]
- **是否需要调整计划**: [Y/N + 原因]
```

### 5.2 图谱迭代

当搜索中发现的新信息与 Step 1 的初始图谱不一致时，**必须更新图谱**：

```markdown
### 📝 供应链图谱更新（Round X 后）
[列出变化:
 - 新发现的具名上游: ...
 - 初步判断错误并修正: ...
 - 新增的风险节点: ...]
```

### 5.3 收敛判断

上下游维度何时算"查够了"？

| 条件 | 说明 |
|------|------|
| 所有 🔴 命脉节点都有至少一个具名实体或量化数据 | ✅ |
| 所有 🟡 重要节点都已尝试≥2种方法 | ✅ 或标记为 stalled |
| 图谱已经过至少1次迭代更新 | ✅ |
| 对信息缺口有明确说明和影响评估 | ✅ |

## 6. Step 5：集中度与风险评估

### 6.1 上游集中度

基于收集到的数据计算/估算：

| 指标 | 计算方式 | 风险阈值 |
|------|---------|---------|
| 供应商集中度 | Top N 供应商占采购总额比 | Top3 > 50% = 🔴高风险 |
| 单一依赖度 | 最大单个供应商占比 | > 30% = 🔴高风险 |
| 替代可行性 | 切换供应商的成本和时间 | > 6个月 = 🟡中风险 |

### 6.2 下游集中度

| 指标 | 计算方式 | 风险阈值 |
|------|---------|---------|
| 客户集中度 | Top N 客户占总收入比 | Top3 > 50% = 🔴高风险 |
| 渠道集中度 | 单一渠道类型占比 | > 70% = 🔴高风险（如KK对购物中心的依赖） |
| 区域集中度 | 单一区域收入占比 | > 50% = 🟡中风险 |

### 6.3 综合风险评级

```markdown
## 供应链风险评估

### 上游风险: [高/中/低]
- 主要风险点: [具体说明]
- 数据支撑: [引用来源]

### 下游风险: [高/中/低]
- 主要风险点: [具体说明]
- 数据支撑: [引用来源]

### 结构性风险: [高/中/低]
- 话语权分布: [谁强谁弱]
- 薄弱环节: [最容易出问题的点]
- 外部冲击敏感度: [经济下行/政策变化/疫情等的影响]
```

## 7. 工具箱（参考，非规定）

> 以下是可供选择的发现手段。Agent **可以**使用这些工具，也**可以**根据自己的判断采用其他方法。
> 唯一要求：如果使用了工具箱以外的方法，需在执行记录中说明。

### 7.1 发现工具一览

| 编号 | 工具名称 | 最适合的场景 | 典型Query思路 | 注意事项 |
|------|---------|-------------|--------------|---------|
| **S01** | **招股书/年报/公告挖掘** | 上市/拟上市/大型企业 | `"公司名" 招股书 供应商 客户 前五大 年报` | **信息质量最高的单一来源**，优先使用 |
| **S02** | **招投标反推** | B2G/B2B/项目型企业 | `"公司名" 中标 成交 公告 site:ccgp.gov.cn` | 对纯C端零售无效，不要浪费时间 |
| **S03** | **股权/投资/子公司追踪** | 所有企业（通用性好） | `"公司名" 子公司 供应商 投资 参股` | 可发现自营供应链设施 |
| **S04** | **新闻/合作/伙伴提取** | 所有企业（通用性好） | `"公司名" 合作 供应商 战略合作 入选` | 注意区分真实合作与PR吹嘘 |
| **S05** | **商场/渠道入驻分析** | 线下零售/餐饮/体验业态 | `"{商场/商圈}" 入驻品牌 主力店 租户` | **零售业特有** |
| **S06** | **加盟/招商信息挖掘** | 连锁/加盟业态 | `"公司名" 加盟 招商 条件 品牌` | 可获知渠道结构和稳定性 |
| **S07** | **诉讼/仲裁文书挖掘** | 所有企业（意外有效） | `"公司名" 诉讼 判决 买卖合同 欠款` | **闭店/欠款/违约事件在此暴露** |
| **S08** | **供应链/行业展会** | 制造业/零售业 | `"{行业}" 供应商大会 展会 参展商` | 可获知行业级供应链玩家 |
| **S09** | **招聘信号反向推断** | 快速增长/特定岗位 | `"城市" {岗位} 招聘 公司` | 正在招采购/供应链管理的人=有供应链活动 |
| **S10** | **商标/专利/认证共现** | 制造业/品牌密集型 | `商标/认证 "{关键词}" 持有人` | OEM/ODM关系有时可通过商标授权发现 |
| **S11** | **社交媒体/点评挖掘** | C2C消费/线下门店 | `"{品牌}" 评价 闭店 欠款 租金` | **一线真实反馈，渠道风险的黄金来源** |
| **S12** | **行业协会/会员名单** | 传统行业/制造业 | `"{行业}协会 会员单位 理事"` | 行业生态参与者地图 |
| **S13** | **地理邻近/园区分析** | 制造业/物流 | `"{园区/地址}" 企业 入驻 租户` | 共享物流/园区的企业可能有供应链关系 |
| **S14** | **海关/贸易数据** | 进出口/跨境电商 | `"{公司名}" 进出口 HS编码 海关` | 可获知跨境供应链伙伴 |

### 7.2 使用建议（非强制）

以下是一些典型的场景-工具匹配建议，Agent 可参考但不必严格遵守：

| 场景 | 推荐优先使用的工具 | 理由 |
|------|------------------|------|

| SaaS/互联网 | S04+S09+S03 | 合作伙伴+招聘信号+投资版图 |
| 跨境电商 | S14+S01+S04 | 海关数据+年报+新闻 |

### 8.2 报告中的体现

供应链分析在报告中应包含：
1. **供应链结构图谱**（Step 1 的输出，可简化呈现）
2. **关键上下游实体清单**（具名的，带证据来源）
3. **集中度数据**（量化的，标注精确/推算）
4. **风险评估**（高/中/低 + 具体风险点）
5. **信息缺口声明**（诚实地标注未获取到的信息及其影响）

## 9. 与竞对维度的交叉关联

### 双向利用

| 方向 | 方法 |
|------|------|
| 供应链 → 竞对 | 同一供应商的其他客户 = 间接竞对候选；同一商场的其他租户 = 渠道竞对 |
| 竞对 → 供应链 | L1竞对的公开供应商名单 → 目标企业可能使用相同供应商；L1竞对的公开客户 → 目标企业的潜在客户池 |

### 交叉标记

同时在 `supply_chain.json` 和竞对相关JSON中用 `cross_refs` 字段记录双重身份的实体。

## 10. ⭐ 下游客户深挖策略（v9.5 新增）

> **常见问题**：上游供应商容易通过产品/技术栈推断（通常3-5家且具体），
> 但下游客户往往只搜到泛称（"某运营商""某超算中心"），缺乏具体名称和合同细节。
> 
> **深挖方法**（按优先级排序）：

| 方法 | Query 示例 | 预期产出 |
|------|-----------|---------|
| **招投标中标记录** | `"{公司名}" 中标 招标 2024 2025` | 具体项目名+金额+采购方 |
| **官网案例/新闻** | `"{公司名}" 案例 客户 成功` | 官方披露的客户名单 |
| **合作伙伴/生态** | `"{公司名}" 合作伙伴 生态 方案` | 渠道伙伴+联合方案中的隐含客户 |
| **行业展会/奖项** | `"{公司名}" 获奖 项目 2025` | 获奖项目往往关联具体客户 |
| **竞对公开客户反推** | `"{竞对名}" 客户 案例 存储` | 同赛道企业的客户池有重叠 |

**质量要求**：
- 下游客户不能只有行业泛称（如"金融行业""运营商"）
- 至少 **3个下游客户** 有具体公司名称 + 合作性质
- 如果确实只能找到泛称，必须在报告中标注 **"下游客户：信息受限，仅能确认X个方向"**
