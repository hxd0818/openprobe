# 实体分层处理策略与种子池管理 ⭐ v9.5

> **核心问题**：每轮迭代都会发现新实体，是否全部按目标公司同等深度调查？
> **答案**：否。全面铺开导致指数爆炸（图遍历陷阱），必须分层处理。
> **本文件定义**：种子池管理（生命周期/选种/扩散）+ 实体分层策略（Tier 1/2/3）+ 与迭代循环的集成点。

## 1. 种子池管理规范

### 1.1 种子生命周期

```
新建（Phase 0 或某轮发现）
  ↓
待探索（explored=false, confidence 0.3-0.7）
  ↓ ← Agent 在 Step ② 中选中
探索中（本轮以该种子为核心设计 query）
  ↓ ← Step ⑤ 标记
已探索（explored=true, explored_round=N）
  ↓
  ┌─ 产出了新发现 → confidence 提升，可能触发新一轮扩散
  └─ 无新发现 → 保持原置信度，降低后续优先级
```

### 1.2 选种策略（Agent 自主决定，参考以下优先级）

| 优先级 | 种子特征 | 理由 |
|--------|---------|------|
| P0 | 🔴命脉节点相关种子（供应链诊断/资本诊断中标为🔴的） | 影响最大 |
| P1 | 高置信度(conf≥0.6) + 未探索 | 成功率最高 |
| P2 | 多源出现（被≥2个独立搜索结果提及） | 可信度高 |
| P3 | 异常信号（高管兼任/同地址/诉讼当事人） | 可能发现隐藏关联 |
| P4 | 低置信度/冷门 | 兜底广度 |

### 1.3 种子池健康度指标

Agent 在 Step ② 时应检查：

```
□ 种子枯竭风险：unexplored 种子 < 5个？
    └─ 是 → 需要回到泛化搜索补充新种子
    
□ 维度覆盖均衡：各维度的 unexplored 种子数是否严重不均？
    └─ 是（如竞对有20个未探索而资本只有2个）→ 补充薄弱维度
    
□ 死种过多：是否有大量 explored 但 confidence 仍 < 0.5 的种子？
    └─ 是 → 这些方向可能信息不可得，考虑标记为 stalled
```

### 1.4 扩散模式

种子扩散不是自动的，而是 **Agent 在设计 query 时的思维方式**：

```
选中种子 S（如"陈世欣"）
  → Agent 思考："关于这个实体我还不知道什么？"
  → 设计能回答这个问题的 query
  → 搜索结果中发现新实体 T（如"某某投资公司"）
  → T 作为新种子归入种子池
  → 下一轮可能选中 T 继续扩散

这就是"扩散"——种子→搜索→新种子→新一轮搜索
```

## 2. 实体分层策略 ⭐核心

### 2.1 为什么不分层？

**图遍历的指数爆炸**：

```
第1轮：目标公司 → 发现 19 个新实体（8竞对+6上游+5下游）
  ↓ 如果每个都做完整调查
第2轮：19 × 各发现 ~10 个关联 = 190 个新实体
第3轮：190 × 10 = 1,900
...→ 搜索轮次/token/时间 全部失控
```

**大部分实体的深度调查价值极低**：
- 上游SSD芯片厂的关注点跟目标公司完全不同
- 下游某地移动分公司的独立调查无决策价值
- 只有 L1 直接竞对值得同等深度投入

### 2.2 三层定义

```
┌──────────────────────────────────────────────────────────┐
│                    实体分层处理矩阵                         │
├──────────┬──────────────┬─────────────┬──────────────────┤
│   层级     │   处理方式     │   产出物       │   典型数量/调查  │
├──────────┼──────────────┼─────────────┼──────────────────┤
│          │              │             │                  │
│ Tier 1   │ 独立完整调查   │ 子目录+完整   │  2~5 家          │
│ 深挖     │ （同目标公司） │ profile      │  (仅L1直接竞对)    │
│          │              │ +分析报告     │                  │
│          ├──────────────┼─────────────┼──────────────────┤
│          │              │             │                  │
│ Tier 2   │ 聚焦快照      │ entity_pool  │  5~10 个         │
│ 快照     │ （2-3个query）│ 丰富条目     │  (L2竞对/关键供    │
│          │ 不跑完整迭代   │ +维度JSON详情 │   应商客户/领投方) │
│          │              │ +一段分析     │                  │
│          ├──────────────┼─────────────┼──────────────────┤
│          │              │             │                  │
│ Tier 3   │ 仅记录存在    │ entity_pool  │  不限（其余所有）  │
│ 占位     │ （名称+类型）│  基础条目     │                  │
│          │              │ +单一来源     │                  │
│          │              │             │                  │
└──────────┴──────────────┴─────────────┴──────────────────┘
```

### 2.3 自动分级规则（Agent 在 Step ④ 执行）

当新实体被发现时，按以下决策树自动分级：

```
发现新实体 E
  │
  ├─ E 是公司类型？
  │   ├─ 是 → 继续判断
  │   └─ 否（person/product/org）→ 默认 Tier 3（除非是关键人物→Tier 2）
  │
  ├─ E 已在 entity_pool 中？
  │   ├─ 是 → 升级 tier（多源发现=更高价值信号），不降级
  │   └─ 否 → 新建，继续判断
  │
  ├─ E 是 L1 直接竞对？（产品高度重叠+直接争夺同一客群）
  │   ├─ ✅ 且满足以下任一条件：
  │   │   · 目标公司是新进入者/挑战者（需深度理解巨头）
  │   │   · 用户明确要求竞对深度对比
  │   │   · 该竞对公开信息极其丰富（如上市公司）
  │   └─ → **Tier 1**
  │
  ├─ E 属于以下类别之一？
  │   · L2 间接竞对
  │   · 前3大上游供应商（按营收贡献或战略重要性排序）
  │   · 前3大下游客户（同上）
  │   · 领投方/控股股东（非自然人）
  │   └─ → **Tier 2**
  │
  └─ 其余所有实体
      └─ → **Tier 3**
```

### 2.4 各层的具体操作规范

#### Tier 1 — 深挖（独立完整调查）

**触发条件**（必须**全部**满足才自动触发，否则降为 Tier 2）：

| # | 条件 | 检查方式 |
|---|------|---------|
| T1-1 | 是 L1 直接竞对 | competitor-research.md Step 2 输出 |
| T1-2 | 目标公司处于挑战者/新进入者位置 | business_boundary 判断 |
| T1-3 | 当前已完成的 Tier 1 数量 < 5 | entity_pool 统计 |
| **T1-4** | **当前轮次 ≥ 3（Phase 0 不计入）** | **⭐v9.5新增：防止过早分心** |

> **⭐v9.5 时机规则**：Tier 1 深挖**不得在 Phase 0 或第1-2轮执行**。
> 原因：前两轮的目标是**建立目标公司自身的完整画像**，
> 过早跳去调查竞对会导致目标公司自身信息不足、后续对比缺乏基准。
> 正确时机：第3轮及之后，当目标公司基础画像置信度 ≥ 0.7 时，
> 可在迭代间隙插入 Tier 1 调查（不占用整轮，而是用 2-3 个聚焦 query 完成）。

**操作流程**：

```
1. 在 data/{目标公司}/competitors/{E公司名}/ 创建子目录
2. 对该实体运行 Phase 0（百科 + 泛化搜索 2-3 query）
3. 生成该实体的 profile.json（精简版，不含完整螺旋迭代）
4. 重点收集（仅聚焦以下维度，不全量展开）：
   ├── 基础画像（规模/员工/营收/阶段）
   ├── 核心产品线（与目标公司的正面战场在哪）
   ├── 近期动态（融资/并购/大事件 18个月内）
   └── 标注数据质量（时效等级+来源）
5. 将产出写入 competitors.json 的 l1_profiles_ref
6. 在主调查的后续轮次中，可针对该竞对补充 1-2 轮聚焦搜索
7. 最终报告中，该竞对获得独立段落（见 report-writing-guide.md §3 的 2.2节）
```

**成本估算**：每个 Tier 1 实体约增加 2-3 轮搜索 + ~8K token 分析

#### Tier 2 — 快照（聚焦查询）

**操作流程**：

```
1. 在主调查的某一轮中，为该实体分配 2-3 个聚焦 query
   （不单独开一轮，混入正常迭代中）
   
2. Query 设计原则（每个回答一个关键问题）：
   ├── "{E公司名} 营收 规模 员工 最新"        → 规模快照
   ├── "{E公司名} 主营业务 产品 核心竞争力"     → 业务快照
   └── "{E公司名} 融资 股东 投资 {最近2年}"     → 资本快照
   
3. 提取结果后，丰富 entity_pool 中该实体的条目：
   ├── 补充 key_facts（法人/成立/注册资本）
   ├── 补充 scale（营收区间/员工数）带时效标注
   ├── 补充 evidence[]（追加新的来源）
   └── 提升 confidence 至 0.6-0.75
   
4. 如果该实体属于 supply_chain/capital 维度，
   同步更新对应 JSON 中的 detail 字段
```

**成本估算**：每个 Tier 2 实体约增加 ~1K token（仅分析，不额外增加搜索轮次）

#### Tier 3 — 占位（最小记录）

**操作**：在 Step ⑤ 正常去重+入库即可，无需额外操作。

```json
// entity_pool 中的 Tier 3 最小记录
{
  "id": "s042",
  "name": "某某零部件有限公司",
  "type": "company",
  "tier": "T3",
  "directions": {"primary": "upstream", "confidence": 0.25},
  "evidence": [{"source": "search_generic", "source_detail": "泛联信息供应商列表提及", "found_at_round": 2}],
  "dimensions_active": ["supply_chain"],
  "notes": "Tier 3占位，未深入调查"
}
```

### 2.5 动态升级机制

实体 tier 不是固定的——随着调查推进可以**升级，不降级**：

```
Tier 3 → Tier 2 条件：
  · 被第2个独立来源提及（多源出现信号）
  · 与目标公司存在非显而易见的关联（如共同投资人/高管兼任）
  
Tier 2 → Tier 1 条件：
  · 重新评估后被认定为 L1（初始判断可能有误）
  · 用户明确要求深挖该实体
  
注意：Tier 1 → 不会降级（一旦投入就保持）
Tier 2 → 通常不降级（除非发现是误判实体）
```

## 3. 与迭代循环的集成

### 3.0 ⭐ 实体池同步铁律（每轮必须执行）

> **问题**：维度JSON（competitors/supply_chain/capital/people）中写入的实体，
> 如果不回写 entity_pool，会导致实体池不完整、去重失效、Tier升级无法追踪。
> 
> **铁律**：在 **Step ⑨ 写回 status.json 之前**，Agent 必须执行一次**实体池同步**：
> ```
> 对每个维度JSON（competitors / supply_chain / capital / people）：
>   遍历其中所有 name 字段 → 检查 entity_pool 是否已存在
>     ├─ 已存在 → 跳过（或追加 dimensions/evidence）
>     └─ 不存在 → 新增一条 Tier 3 记录（name + type + source=该维度JSON + tier=T3）
> 
> 同步后检查：
>   □ entity_pool 实体数 ≈ 各维度JSON实体去重总和？（允许≤10%偏差）
>   □ 差异实体是哪些？为什么没入库？
> ```
> **违反后果**：entity_pool 不完整 → 最终报告的"发现N个实体"数字不可信。

### 3.1 修改点：Step ④ 分析整合

在 `methodology.md` Step ④ 的流程中，插入一个分支判断：

```
原流程：
  逐条阅读搜索结果 → 发现新实体 → 准备加入种子池（Step ⑤）

修改后：
  逐条阅读搜索结果 → 发现新实体 E
    → 判断 tier（§2.3 决策树）
    → 去重检查
    → 加入种子池（带 tier 标记）
    → 如果是 Tier 1：
        → 排入 independent_research_queue（见 status.json 扩展字段）
    → 如果是 Tier 2：
        → 标记 pending_snapshot=true
        → 在本轮或下轮分配聚焦 query
    → 如果是 Tier 3：
        → 正常入库即可
```

### 3.3 收敛条件调整

Tier 1 和 Tier 2 的处理状态**纳入收敛判断**：

```
新增收敛前置条件 C5（可选，仅在存在 Tier 1 实体时激活）：
  C5: 所有 Tier 1 实体的独立调查已完成（status=done）
      且所有 Tier 2 待快照实体已完成快照（t2_pending 为空）
      
如果 C5 未满足但 C1-C4 已满足：
  → 可以生成报告，但在封面标注 
    "⚠️ 部分关联实体调查进行中（N家竞对待深挖）"
```

## 4. 跨调查复用

### 4.1 同赛道复用

```
场景：已调查过 A 公司（发现了竞对 B/C/D/E）
      现在要调查 B 公司

复用策略：
  1. 检查 data/A公司/competitors/B/ 是否存在独立 profile
     ├─ 存在且质量良好（≥T2级别）→ 直接复用作为 B 的 Phase 0 种子
     ├─ 存在但较旧（>6个月）→ 复用基础上增量搜索最新动态
     └─ 不存在 → 正常从零开始
     
  2. A 公司的 entity_pool 中关于 B 的条目 → 导入 B 的初始种子池
  3. A 公司的 supply_chain.json 中 B 的上下游关系 → 参考线索
```

### 4.2 存储约定

```
data/
├── {目标公司A}/
│   ├── competitors/
│   │   ├── {竞对B}/           ← Tier 1 独立目录
│   │   │   ├── profile.json
│   │   │   ├── raw/           ← 仅该竞对的搜索原始数据
│   │   │   └── company_profile.md  ← 可选：独立简报
│   │   └── {竞对C}/
│   ├── status.json
│   ├── entity_pool.json       ← 所有实体的统一索引（含tier标记）
│   └── ...
│
└── {目标公司B}/               ← 下一次调查
    └── （可引用 A 的 competitors/ 数据）
```

## 总结：分层策略速查卡

```
发现新实体 → 判断类型 → 分配层级 → 执行对应操作

  L1竞对 + 条件满足？  → Tier 1 → 独立子目录 + 完整profile + 报告段落
  L2竞对/关键供客/投资？ → Tier 2 → 2-3聚焦query + 丰富entity_pool条目
  其他？                → Tier 3 → 记录名称+类型+来源，防止重复

核心约束：
  - Tier 1 总数 ≤ 5（默认上限，用户可调）
  - Tier 2 不额外增加搜索轮次（混入主迭代）
  - 只升不降（多源发现自动升级）
  - 纳入收敛判断（C5条件）
```
