# PIR（优先情报需求）模板与收敛框架

> 本文件定义 OSINT 企业调查的 **PIR 驱动收敛机制**。
> 核心思想：收敛的标准是"关键问题是否都已回答"，而非"搜到了多少实体"或"跑了几轮"。
> 方法论来源：NATO 情报循环模型 + Rae Baker "Knowing When to Stop in OSINT"

## 一、PIR 四层结构

### Layer 0：用户指定关注点（最高优先级）

用户在发起调查时可指定一个或多个特别关注的方面。这些自动成为 CRITICAL 级 PIR，且 target_confidence 设为 0.9（高于基线）。

```json
{
  "id": "PIR-USER-01",
  "source": "user_specified",
  "question": "IPO前景如何？四次递表失效的根本原因是什么？",
  "priority": "CRITICAL",
  "target_confidence": 0.9,
  "status": "open",
  "current_confidence": 0.0,
  "added_at": "phase_0"
}
```

**触发方式**：用户说"帮我调查XX公司，特别关注它的YY和ZZ" → 自动生成对应 PIR。
**无指定时**：Layer 0 为空，直接进入 Layer 1。

---

### Layer 1：基线 PIR 模板（所有调查必须包含）

> 这些是任何企业 OSINT 调查都必须回答的核心问题。
> Agent **不能删除** Layer 1 的 PIR，只能追加或在 Layer 2 中调整权重。

| PIR ID | 问题 | 优先级 | 目标置信度 | 关联维度 | 最少证据信号 |
|--------|------|--------|-----------|---------|-------------|
| PIR-BASIS | 这家公司是谁？做什么的？多大规模？处于什么阶段？ | CRITICAL | 0.85 | profile | 营收/员工/门店/业务模式/行业地位 |
| PIR-COMPETITION | 谁在和它抢生意？格局怎样？相对位置？ | CRITICAL | 0.80 | competitor | L1竞对名单+规模+战略动向 |
| PIR-SUPPLY_CHAIN | 上游谁供货？下游谁买单？依赖度和风险？ | HIGH | 0.75 | supply_chain | 具名上游≥5家 + 下游≥5家 + 集中度评估 |
| PIR-CAPITAL | 钱从哪来？谁能活多久？融资/IPO前景？ | HIGH | 0.75 | capital | 股权结构 + 融资历史 + 投资人意图 |
| PIR-PEOPLE | 谁在做决策？背景如何影响公司走向？ | HIGH | 0.75 | people | 决策者≥5人 + 竞对CEO对比 + 投资人代表 |

**证据信号说明**：每个 PIR 下列出的是"最少需要获取什么才能认为该 PIR 达标"。不是必须全部拿到，但缺得太多就不能标记为已满足。

---

### Layer 2：Agent 按企业类型实例化

在 Phase 0（种子初始化）完成百科查询后，Agent 根据**目标企业的行业和特征**对 PIR 做实例化调整：

#### 2.1 行业适配追加表

| 企业类型 | 追加 PIR | 示例问题 | 优先级 |
|---------|----------|---------|--------|
| **零售消费** | PIR-CHANNEL | 渠道健康度？闭店/加盟商稳定性？坪效趋势？ | HIGH |
| **零售消费** | PIR-BRAND | 品牌力如何？自有品牌vs第三方占比？ | MEDIUM |
| **制造业** | PIR-PRODUCTION | 产能/代工关系？供应链安全？OEM/ODM依赖？ | HIGH |
| **SaaS/互联网** | PIR-TECH | 技术栈/产品壁垒？用户留存/增长？DAU/ARR？ | HIGH |
| **SaaS/互联网** | PIR-DATA | 数据安全/合规？隐私政策？ | MEDIUM |
| **金融** | PIR-COMPLIANCE | 监管合规？牌照？资本充足率？ | HIGH |
| **G端服务/咨询** | PIR-GOV | 政府关系？招投标能力？资质/认证？ | HIGH |
| **拟上市/IPO中** | _(强化 PIR-CAPITAL)_ | target_confidence 提升至 0.90 | — |
| **有诉讼/纠纷记录** | PIR-LITIGATION | 未决诉讼？合规风险？对公司影响？ | HIGH |
| **近期有大额M&A** | PIR-M&A | 收购标的？整合风险？资金来源？ | MEDIUM |

#### 2.2 裁剪规则

- **禁止静默删除** Layer 1 的任何 PIR
- 如某 PIR 确实不适用（如非上市公司无需关注 IPO），必须：
  - 保留该 PIR 但降低 priority 为 LOW
  - 在 `note` 字段写明原因：`"目标为非上市公司，IPO相关PIR降级"`
  - 用基础信息替代（如改为"融资历史和资金充裕度"而非"IPO进展"）

---

### Layer 3：动态追加（迭代过程中发现的新问题）

当调查中发现意外线索时，Agent 可动态追加新的 PIR：

**触发条件**（满足任一即考虑追加）：
- 发现未在现有 PIR 覆盖范围内的重大风险信号（诉讼、监管处罚、重大人事变更）
- 发现可能改变核心判断的新信息（如竞对突然并购、投资人退出）
- 某个维度的深度足以拆分为独立子问题

**追加格式**：

```json
{
  "id": "PIR-DYN-01",
  "source": "dynamic_discovery",
  "discovered_at_round": 3,
  "trigger_evidence": "搜索中发现KK集团与名创优品存在商标不正当竞争诉讼",
  "question": "商标诉讼的具体情况？进展？可能的赔偿/禁售风险？",
  "priority": "HIGH",
  "target_confidence": 0.70,
  "status": "open",
  "current_confidence": 0.0,
  "parent_pir": "PIR-COMPETITION"  // 从哪个父PIR派生
}
```

**限制**：
- 单次调查最多动态追加 **5 个** PIR（防止发散）
- 动态 PIR 的 priority 不能超过 CRITICAL（除非关联用户指定的 Layer 0）
- 必须记录 `trigger_evidence`（什么发现触发了这个新问题），不可凭空追加

## 二、PIR 置信度评估方法

### 2.1 置信度等级定义

| 置信度 | 含义 | 判断标准 |
|--------|------|---------|
| **0.90 - 1.00** | 高度确信 | 多源交叉验证 + 原始文件佐证（财报/招股书/公告） |
| **0.75 - 0.89** | 较可信 | 多个独立来源一致 + 无矛盾信息 |
| **0.50 - 0.74** | 初步判断 | 有一定证据但来源单一或需进一步验证 |
| **0.25 - 0.49** | 弱信号 | 仅零散提及，无法独立支撑结论 |
| **0.00 - 24** | 无信息 | 完全未获取到相关信息 |

### 2.1.1 ⭐ 推算等级折扣系数（v9.3升级 — 见 `data-quality-gate.md` §2.5.C）

> PIR 的最终有效置信度 = 原始置信度（基于证据强度）× 推算等级系数
>
> **⭐v9.3 变更**：从4级时效分级升级为5级推算等级（含碎片推算和旧数据推算区分）。
> 旧数据不可直接使用，必须经推算后才可纳入 PIR 判断。

| 数据类型 | 折扣系数 | 说明 |
|---------|---------|------|
| 🟢 FRESH (≤6月) | **1.00** | 直接获取的最新年完整数据，不打折 |
| 🟡 CURRENT (6-18月) | **0.95** | 去年完整财年数据，轻微打折 |
| **~ 碎片推算 (Level 2)** | **0.85** | ⭐v9.3新增：用新数据碎片推算的完整值 |
| **~ 旧数据推算 (Level 3)** | **0.65** | ⭐v9.3新增：用🟠旧数据+推算因子映射到当前年份 |
| 🔴 OBSOLETE (>36月) | **0.40** | 仅历史基线，几乎不可用于当前判断 |

**与 v9.2 的关键差异**：
- v9.2：🟠 STALE 可直接使用（×0.75）
- v9.3：🟠 旧数据**不可直接使用**，必须经 Level 3 推算后以 ×0.65 纳入
- 新增 Level 2 碎片推算（×0.85），鼓励 Agent 用碎片拼出新数据而非直接降级

**执行时机**：每轮更新 PIR current_confidence 后（methodology.md Step ⑥），检查该 PIR 所依赖的核心数据的推算等级，应用折扣。如果打折后 effective_confidence < target_confidence，Agent 必须：
1. **优先尝试 L1 搜索更新的完整数据**（带时间锚定 query，≥2次）
2. 或 **L2 碎片 hunt**（≥3次不同角度搜索碎片数据）
3. 或 **L3 旧数据+推算因子**（寻找增速/规模变化等桥梁信息）
4. 或将该 PIR 标注为 `stalled` / 受限并说明原因
5. 完整四级递进流程见 `data-quality-gate.md` §2.5.A

### 2.2 每个 PIR 的置信度更新规则

每轮迭代结束时，Agent 对每个活跃 PIR 执行：

```
本轮是否获取了与该 PIR 相关的新证据？
├─ 是 → 提升 confidence（幅度取决于证据质量）
│   ├─ 获得"决定性证据"（如招股书原文）→ +0.20~0.30
│   ├─ 获得"强佐证"（如权威媒体报道）→ +0.10~0.15
│   ├─ 获得"弱信号"（如论坛/社交媒体提及）→ +0.03~0.08
│   └─ 发现矛盾信息 → 不提升，标注矛盾待解决
│
└─ 否 → confidence 不变
    └─ 连续 2 轮无任何提升 → 标注 "stalled"
```

### 2.3 PIR 状态机

```
open → in_progress → satisfied / stalled / abandoned / n/a
                    ↓
              (可被新证据重新激活)
```

| 状态 | 含义 | 条件 |
|------|------|------|
| `open` | 尚未开始针对此 PIR 采集 | 初始状态 |
| `in_progress` | 已开始采集，但置信度未达标 | 有至少一轮搜索与此 PIR 相关 |
| `satisfied` | ✅ 已达到 target_confidence | current ≥ target |
| `stalled` | ⚠️ 连续多轮无进展 | 连续2轮无提升 + 已尝试≥2种策略 |
| `abandoned` | ❌ 确认无法获得（信息不存在） | 多角度搜索均无结果，标注"信息缺口" |
| `n/a` | ➖ 不适用 | Layer 2 裁剪降级的 PIR |

## 三、基于 PIR 的收敛判断流程

### 3.1 每轮结束时的检查清单

```
□ 更新所有活跃 PIR 的 current_confidence
□ 标记本轮是否有任一 PIR 提升（有效轮判定）
□ 检查是否有 PIR 进入 stalled 状态
□ 如有新发现触发 Layer 3 → 追加动态 PIR
□ 记录到 round_logs/round_N.json
```

### 3.2 收敛前置条件（必须全部满足才可宣布收敛）

| # | 条件 | 说明 |
|---|------|------|
| C1 | 所有 CRITICAL PIR 状态 ∈ {satisfied, n/a} | 核心问题必须有答案 |
| C2 | 所有 HIGH PIR 状态 ∈ {satisfied, stalled, abandoned, n/a} | 重要问题要么答了要么确认答不了 |
| C3 | 已执行"最后验证轮" | 换角度广扫确认无遗漏 |
| C4 | 有效轮次 ≥ **6** | ⭐v9.5从4上调：确保做了足够工作，轮次是下限护栏非收敛目标 |

### 3.3 最后验证轮（Final Validation Sweep）

在 C1+C2 即将满足时，**必须**执行一轮换角度扫描：

```bash
# 最后验证轮的 query 设计原则：
# 1. 用已发现的实体名反查目标公司
# 2. 用时间维度刷新（加上最新年/季度关键词）
# 3. 用地域维度补充（具体城市/区域）
# 4. 检查跨维度关联是否有遗漏
# 5. 快速验证所有 satisfied PIR 的关键事实是否有矛盾
```

**结果处理**：
- 有新发现 → 重置相关 PIR 的 stalled 状态，继续迭代
- 确认无新发现 → 正式收敛

### 3.4 安全阀

| 条件 | 动作 | 输出 |
|------|------|------|
| 达到 **15轮** 仍未收敛 | 强制暂停 | 生成"阶段性报告"，标注哪些 PIR 未满足及原因 |
| 连续 **4轮无效轮**（非 stalled，是真的没有任何输出） | 触发策略反思 | 检查 query 设计是否有问题，考虑换搜索引擎或调整方向 |
| 所有 PIR 已 satisfied 但有效轮 < 4 | 继续迭代 | 即使问题都答了，也要做验证轮确保没有假阳性 |

### 3.5 收敛后的动作

收敛确认后 → 进入报告生成阶段（按 `references/output-specs.md` 的规范）。

报告中必须包含 **PIR 履行情况摘要**：

```markdown
## PIR 履行情况

| PIR | 问题 | 优先级 | 目标置信度 | 最终置信度 | 状态 |
|-----|------|--------|-----------|-----------|------|
| PIR-BASIS | 公司基本画像 | CRITICAL | 0.85 | 0.92 | ✅ satisfied |
| PIR-COMPETITION | 竞争格局 | CRITICAL | 0.80 | 0.88 | ✅ satisfied |
| PIR-SUPPLY_CHAIN | 产业链 | HIGH | 0.75 | 0.71 | ⚠️ stalled (下游渠道数据不足) |
| ... | ... | ... | ... | ... | ... |
```

这既是给读者的质量声明，也是 Agent 自我审计的工具。

## 四、与传统"数实体/数轮次"方法的对比

| 维度 | 旧方法（实体计数驱动） | 新方法（PIR 需求驱动） |
|------|---------------------|---------------------|
| 收敛信号 | "连续N轮无新实体" 或 "达到M轮" | "所有关键问题都已回答" |
| 成功度量 | 实体数量 / 轮次数 | PIR 置信度 / 决策支持能力 |
| 方向选择 | 按维度缺口选 | 按**最低置信度 PIR** 选 |
| 过度收集风险 | 高（没明确目标就停不下来） | 低（问题答了就停） |
| 收集不足风险 | 高（实体够了但关键问题没回答） | 低（PIR 直接映射到决策问题） |
| 可追溯性 | 只知道搜了几轮 | 每个结论都能追溯到具体 PIR |
| 用户参与感 | 无 | 可通过 Layer 0 注入关注点 |
